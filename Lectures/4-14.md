### rdt3.0 Recap of Packet Flow

**rdt is a stop-and-wait protocol, it sends one packet at a time**

**(a) No loss**

Each packet is being sent and the receiver has no problem acknowledging the packets.

**(b) Packet loss**

If a packet is lost in the channel, you wait for an amount of time, then resend the packet.

**(c) ACK loss**

If the acknowledgement is lost, resend the packet.

**(d) Premature timeout/delayed ACK**

### Example:

- L = 8000 bits (1 packet)
- R = 1 Gbps (rate at which sending out. 1 Gbps = 10⁹ bps)

**What is the time it would take to send out this one packet?**

L / R = 8000/10⁹ = 8 microseconds

**If RTT = 30 msec, then the time it would take to send out one packet and receive the ACK:**

RTT/2 + L/R = 15 msec + 8 μsec ≈ 15.008 msec

This is too long for sending out one packet.

---

### Pipelining:

You send out one packet and you receive an ACK, this is an RTT. In pipelining you don't have to wait for ACK 0 to send out a new packet.

**Utilization = (L/R) / (RTT + L/R) × N**

### Go-Back-N: Sender

We can't overwhelm the receiver by blatantly pushing packets into the network.

The window in this protocol only moves forward if the oldest unacknowledged packet is now acknowledged.

Always send in order. If we acknowledged packet 10 and it's been received, ACK = 10, this means every packet before packet number 10 has been received by the receiver.

Go-Back-N deals with cumulative ACKs.

### Selective Repeat:

Need individual ACKs for each packet and individual timers because if a packet times out, only send the packet that is missing to the receiver.

### Comparison: Go-Back-N vs Selective Repeat

**Go-Back-N:**
- Cumulative ACK
- Admits packets in-order
- Individual timers → why? → discard all out-of-order packets

**Selective Repeat:**
- Individual ACK
- Buffers packets out-of-order
- Receiver window

For a given packet length L, the time varies with rate: T = L/R. The higher the rate used to transfer data, the less time it takes.

### rdt2.2: Sender Recap

**3 points to remember:**
1. Send out a packet, and expect an ACK, hope to get a positive ACK
2. If we don't get a positive ACK then we get a negative ACK. This means an incorrect or corrupt packet was sent.
3. If this is the case we resend the packet.
