## TCP Reliability, Flow Control, and Congestion Control Part 2

### TCP Flow Control:

**What happens if network layer delivers data faster than application layer removes data from socket buffers?**

**Flow Control:** mechanism that prevents the receiver being overrun by the sender.
- Receiver controls sender, so sender won't overflow receiver's buffer by transmitting too much, too fast

---

### How TCP Implements Flow Control:

Receiver informs the sender how much free buffer space there is, and the sender limits itself to send no more than that amount of data. The limit set by the receiver is **rwnd** (receive window).

- TCP receiver "advertises" free buffer space in rwnd field in TCP header
  - RcvBuffer size set via socket options
  - Many operating systems autoadjust RcvBuffer
- Sender limits amount of unACKed data to received rwnd
- Guarantees receive buffer will not overflow

### TCP Connection Management

Before exchanging data, sender/receiver "handshake":
- Agree to establish connection (each knowing the other is willing to establish connection)
- Agree on connection parameters

The TCP sender and receiver have a number of pieces of shared state that they need to establish before they actually begin communication.

### 2-Way Handshake Scenarios

Client makes a connection request with sequence number x which is acknowledged by the server.

They then both enter the established state after the two-way handshake. The sender sends a piece of data with sequence number x + 1 which is acknowledged by the server, and the connection is closed.
